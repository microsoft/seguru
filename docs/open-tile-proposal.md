# `open_tile()` — reshape_map codegen amortization

## Goal

Reduce epilogue address-arith bloat in per-thread tile kernels without
weakening `reshape_map!`'s memory-safety guarantees.

## Mechanism

Add two companion methods on maps generated by `reshape_map!` (via a new
`MapWithLidOffset` trait in `gpu::chunk`):

```rust
// Full index:    offset + tid-contribution + lid-contribution
fn map(&self, lid, tid) -> (bool, u32);              // existing

// lid-contribution only (no offset, no tid-portion):
fn map_lid_offset(&self, lid) -> (bool, u32);

// offset + tid-contribution (tid-invariant for a given thread):
fn thread_base(&self, tid) -> (bool, u32);
```

Contract (enforced by the macro-generated code, not by user implementations):

```
map(lid, tid).1 == thread_base(tid).1 + map_lid_offset(lid).1
```

Then expose `open_tile(self)` on `GlobalGroupChunk` (`ToScope=Thread` bound).
It consumes the chunk and returns `OpenedTile<'a, T, CS, Map>` which caches
a 64-bit `tile_ptr = data.as_mut_ptr().add(thread_base)` once and indexes via
`tile_ptr.add(map_lid_offset(lid))` on each access.

## Memory safety

Unchanged:

- `open_tile(self)` consumes the chunk, so at most one `OpenedTile` per chunk.
- `OpenedTile<'a, ...>` holds `PhantomData<&'a mut [T]>` which keeps the
  exclusive-borrow lifetime from the underlying slice. No aliasing.
- Each per-access `IndexMut` still AND's `base_ok` (tid validity + struct
  precondition, checked once at open) with `lid_ok` (per-access lid bounds)
  via `assert_ptr`, just like today's `IndexMut`.
- `MapWithLidOffset` is `unsafe` and documents the additivity invariant the
  macro satisfies by construction (two independent per-dim loops: lid-dims
  and tid-dims — symmetric to the existing combined loop, same bounds/reverse
  semantics per dim).

## Measured result (Phase J.4, A100 CC 8.0)

Canary: `gemm_add_relu` (the kernel that triggered Phase I diagnosis).

| Metric                       | Before open_tile | After open_tile |
|------------------------------|-----------------:|----------------:|
| Registers/thread (ptxas -v)  | 168              | 168             |
| `add.s64` per kernel         | 77               | 64              |
| Compare.py speedup vs torch  | 0.39×            | 0.40×           |
| Compute throughput (ncu)     | 35.67%           | (not re-run)    |
| Correctness                  | 12/12            | 12/12           |

**Conclusion:** `open_tile` successfully eliminates per-store recomputation of
the thread base in the epilogue (PTX now shows one shared `%rd22` base across
all 64 stores instead of 8 recomputed bases). That's a real, verifiable codegen
improvement.

However, register count did **not** drop, because the 168-register footprint
is dominated by:

1. The 64-entry `acc[8][8]` f32 accumulator (kept live across the epilogue)
2. Per-tile-row `a_reg[8]` and `b_reg[8]` loaded from shared memory
3. Address-arith temporaries in the **main loop** (not just the epilogue)

The epilogue is ~15% of GEMM runtime, so even a perfect elimination of
epilogue address-arith caps the headline speedup at ~15%. In practice the
observed gain was within noise (0.39× → 0.40×).

## What this confirms / refutes

- **Confirms:** `reshape_map` *can* be amortized without weakening memory safety.
  The `MapWithLidOffset` contract is well-defined and the macro satisfies it.
- **Refutes:** Epilogue-only amortization closes most of the 2.18× gap to
  nvcc on GEMM. The gap is register-pressure-driven, and the pressure
  originates in the main loop (accumulators + hot loads), not the epilogue.

## Next candidates (if pursuing further)

In order of expected ROI:

1. **Accumulator fragmentation.** Today `acc` is an `[f32; 64]` array in local
   memory that the backend may not always promote to registers. Promote via
   per-dim explicit variables, or trust LLVM's scalarreplaggr pass by
   hoisting/splitting the accumulator.
2. **Register-reuse hint to ptxas.** Use `--maxrregcount=128` at the *source*
   via `#[gpu_codegen::maxnreg(128)]` or similar, scoped to the kernel, once
   accumulator fragmentation has dropped spills below ~16 B.
3. **Main-loop `open_tile` on tile_a / tile_b shared reads.** Extends
   `open_tile` to `tile_a.chunk_mut(load_map)` so the per-iteration reload
   reuses a pre-offset shared-mem pointer.

None of those are part of this PR. This PR ships `open_tile` as a clean,
memory-safe primitive that can be reused by follow-up work without further
reshape_map changes.
